FEATURE 1 — BYPASS SUPABASE 1000 ROW LIMIT
--------------------------------------------

Currently the application only loads a maximum of 1000 rows from Supabase because that is the default row limit per query. This means students with many records or bulk result views are incomplete. You must fix this everywhere results or marks data is fetched.

Create a new utility file in the lib folder. This utility must contain a function that fetches all rows from any given Supabase table by using range-based pagination internally. The function should accept the table name and an optional query modifier function so that callers can still apply filters like filtering by student ID or semester or branch. The function should loop in batches of 1000, using the range method on the Supabase query builder, appending results to an accumulator array, and stopping when a batch returns fewer rows than the batch size. It should return the complete array of all rows.

Create a second helper in the same file that does the same but accepts a custom select clause instead of always using select star.

Create a third helper that returns just the count of rows using the head and count exact options on the Supabase query, useful for showing progress or totals.

After creating this utility, search the entire codebase for every place where student marks or results are fetched from Supabase. Replace those calls with calls to this new utility so that every single mark is loaded regardless of how many rows exist. Do not change the Supabase client configuration file itself. Do not change the table structure. Only change the fetch calls.


FEATURE 2 — SEMESTER WISE RESULTS DISPLAY

When a user taps or clicks on a student and sees that student's details, currently all results are shown in a flat list or a single view. You need to add a new section below the existing student details that organizes and displays results semester by semester.

Read the actual Supabase table and identify which column represents the semester number, which represents the subject name, which represents the subject code, which represents marks obtained, which represents maximum marks, and which represents grade if it exists. Use those exact column names throughout. The grouping function should return an array of semester objects, each containing the semester number, an array of subject results for that semester, the total marks obtained, the total maximum marks, the calculated percentage, the number of passed subjects, the number of failed subjects, the total number of subjects, and the overall pass or fail status for that semester. Sort the output by semester number ascending.

Include a helper function that calculates letter grades from percentage if the grade is not already stored in the database. Use a standard grading scale such as O for 90 and above, A+ for 80 and above, A for 70 and above, B+ for 60 and above, B for 50 and above, C for 40 and above, and F for below 40. Include a helper that returns the list of distinct available semesters for a given set of marks data.

Next, create a new component that serves as the semester-wise results viewer. This component receives the raw marks data array, the student name, the student ID, and an optional passing percentage threshold which defaults to 40. Inside this component, use the grouping utility to organize the data. Render a row of semester selector buttons or tabs at the top, one for each available semester plus an All Semesters option. When a semester is selected, display a table showing all subjects for that semester with columns for serial number, subject code, subject name, marks obtained, maximum marks, percentage for that subject, grade, and pass or fail status. Below the table show a footer row with totals. Above the table show a header bar for that semester displaying the semester number, overall percentage, and overall pass or fail badge.

Integrate this component into the existing student detail page by importing it and placing it below all existing content. Fetch the student's marks using the new fetch-all utility created in Feature 1. Pass the loaded marks to this new component. FEATURE 3 — STATISTICS WITH PIE CHARTS
----------------------------------------

For each semester displayed in the semester results viewer, add a statistics section below the marks table. This section should contain three pie charts side by side or in a responsive grid.

In the admin panel, add a new section or page accessible from the admin navigation. Call it Activity Log or Faculty Actions or Audit Log. Provide basic filtering options allowing the admin to filter by faculty member, by action type, by date range, and by entity type. Provide pagination since the log could grow large. Use the fetch-all utility to load all log entries, or implement server-side pagination if the admin panel already uses server-side patterns.

This gives the admin complete visibility into what every faculty member is doing in the system, what operations they are performing, and what data they are changing.
